**本ドキュメントの位置づけ:**
本書は IEC 62304 に基づく医療機器ソフトウェア開発プロセスを、どの順序・どの根拠で進めたかを時系列で記録する。後続プロジェクトが IEC 62304 の進め方を参照する際のお手本となることを目的とする。

## 全体戦略(フェーズ区分)

本プロジェクトは以下のフェーズで進める。各フェーズの意図を先に示すことで、個別ステップの採用根拠が俯瞰できる。

| フェーズ | 目的 | 主成果物 | IEC 62304 条項との対応 |
|---------|------|---------|-----------------------|
| **Phase 1: 分類と計画**(M0 開始期) | 安全クラス決定と開発プロセス全体計画の確立 | SSC、SDP | §4.3(分類)、§5.1(開発計画) |
| **Phase 2: 支援プロセス計画**(M0 基盤整備期) | リスク・構成管理・問題解決・保守の各プロセス計画を確立 | SRMP、SCMP、SPRP、SMP | §7、§8、§9、§6 |
| **Phase 3: インフラ整備**(M0 終了期) | ドキュメント公開・GitHub 運用・補助台帳の整備 | README、Issue/PR テンプレート、CIL、CCB 運用規程、CRR、RMF 初版 | §5.1.9、§5.1.10、§7.1.4 |
| **Phase 4: Inc.1〜Inc.4 実開発**(M1〜M4) | V字を機能インクリメントで一周ずつ実施 | SRS、SAD、SDD、コード、UT/IT/ST 計画・報告 | §5.2〜§5.8 |
| **Phase 5: リリースと保守**(M_final 以降) | 全インクリメント統合、最終試験、v1.0.0 リリース、以降 SMP による保守 | SMS、リリース成果物、RMF 更新 | §5.8、§6 |

Phase 1〜2 の「計画先行」方針を採る根拠:

1. **IEC 62304 §4.3** は「安全クラスは他のすべてのプロセスに先行して決定する」ことを要求する。
2. **§5.1.1** は「開発プロセスを計画する」ことを要求し、SDP は他のすべての成果物から参照される上位計画となる。
3. 支援プロセス計画(SRMP/SCMP/SPRP/SMP)は、Inc.1 の SRS 作成以降で常時参照・運用されるため、**先に用意しないと後続工程が手戻りする**。
4. 先に計画書を完成させることで、Inc.1 以降の実開発時に「どのプロセスに従うか」の判断で悩まなくなる。

## 教訓(随時追加)

- **ID 衝突の早期発見**:Step 6(SPRP 作成)で、テンプレートの `PR-`(Problem Report)と GitHub の `PR`(Pull Request)が混同することに気付いた。支援プロセス計画を連続で書いたからこそ発見できた。ID 体系は複数ドキュメントで使われるため、**最初に固めるか、連続で書いて早期に整合化する** のがよい。
- **計画先行の効用**:Phase 1〜2 で計画書 6 本を連続で作成したことで、以降の工程で「どのプロセスに従うか」の判断で悩む必要がなくなった。学習プロジェクトでも「計画を先に出し切る」のは有効。
- **GitHub ラベルとテンプレートの同時整備**:Step 14a で発覚。Issue テンプレート frontmatter に `labels: change-request` を指定するだけではラベルは作成されない。**Issue テンプレート整備時には対応するラベルを `gh label create` で同時に登録する** 必要がある。Step 9 のチェックリストに「ラベル作成」を追加すべき教訓。`change-request` 以外に `problem-report`、`enhancement` も将来的に必要。
- **プロセス規程の条文ベース参照**:Step 14a 検討時に「24h インターバル」の起算点を「直近の作業日からの経過時間」と誤認し、誤った提案をした。CCB-VIP-001 §5.4 を直接参照し直して「Issue 起票時刻 〜 PR 作成時刻」が正規定義であることを確認、案 P(2 セッション分割)に修正。**プロセス規程は要約や記憶に頼らず、条文を都度参照すること。** 特に時間制約・ID 規則・承認フローは齟齬が出やすい。
- **詳細設計が要求を完成させる**:Step 14a の SDD v0.2 化作業で 4 件の SRS 整合性課題(ISS-V02-001〜004)を発見。**詳細設計の擬似コードを書かない限り「タイムアウト値が未明示」「定常/過渡区別が曖昧」「ファストパス必要性」「期限値未定義」が顕在化しない。** IEC 62304 §5.2.4 / §5.4.4 が要求する「設計工程からの SRS 改訂フィードバック」は、お題目ではなく実工程で必ず発生する。発見時はすぐに新規 CR で SRS を改訂するか、本 CR の追記事項として申し送る運用が必須。
- **CI 失敗を複数ステップ連続で放置するリスク**:Step 14b で発覚。Step 9 の教訓で「CI は本リポジトリで一度も成功していなかった」と記録されていたが、その後の Step 11(SRS v0.1)・Step 12a(RMF v0.2)・Step 12b(CIL v0.2)・Step 12c(SAD v0.1)・Step 13(SDD v0.1)すべてで CI 失敗を放置したまま進行していた。原因は SRS §4.9 表の列数不整合(MD056)という軽微誤記だったが、これを **PR #2 起票時まで見逃し続けた**。3 ステップ連続の累積は監査時に「プロセス外進行」として扱われる懸念がある。**対策:**(1) 各コミット push 後に `gh run list --repo <正しい repo>` で CI 結果を確認、(2) 失敗時は次ステップに進む前に必ず修正、(3) CI 失敗を検知する仕組み(Slack 通知・メール通知・GitHub の通知設定)を Step 9 の補強項目として追加検討。本項目は SPRP の対象(根本原因分類「プロセス誤り」、重大度 Minor)として扱うべきだが、学習プロジェクト特例として Step 14b で即時修正・遡及記録で処理。
- **マージコミット方式の採用判断**:Step 14c で最初の PR マージを実施する際、`--merge` / `--squash` / `--rebase` の 3 択で迷走する可能性があった。**SCMP §3.1 のトレースチェーン「[コミット] → [マージコミット] → [ベースラインタグ] → [リリースタグ]」が merge commit を前提にしている** ことを条文で確認して決定。squash は個別コミット意図の消失、rebase はリニア履歴で分岐・統合の事実が消える点でトレース性が劣る。この判断を Step 14c の根拠として残すことで、後続 PR(CR-0002 以降)でも一貫して merge commit 方式を採用する根拠となる。**プロセス規程の条文を直接参照する習慣**(Step 14a 教訓の再強化)。
- **admin bypass による main 直接 push の扱い**:Step 14c で CRR v0.3 を main に直接 push した際に `Bypassed rule violations: 4 of 4 required status checks are expected` の警告が記録された。この警告の意味は「admin 権限で required status check(CI 4 ジョブ)をスキップして push した」というもので、GitHub の監査ログに永続記録される。**問題の観点は 3 つ:**(1) 品質観点 — CI が実行されないため、編集ミス(MD056・日付書式違反等)を検出できない、(2) 監査観点 — IEC 62304 §8 構成管理で「全 main 変更が検証を通った」ことを示す必要があるが bypass ログが残る、(3) お手本観点 — 後続プロジェクトが「軽微だから bypass」を濫用する懸念。**今回は「マージ後アクションの軽微台帳更新」という限定条件で許容したが、Step 14d/14e の統合コミット化で bypass 回数を 1 回に削減。** 将来対策:CR-0002(規程改訂)と同時に SCMP §4.1.2 に「マージ後アクションの軽微台帳更新は直接 push 許可、ただし事前に `npx markdownlint-cli2@0.13.0` 等でローカル lint を必須化」を明文化することを検討。**教訓の核心:**ブランチ保護ルールの bypass は「緊急避難」として扱い、濫用されない仕組み(監査ログレビュー・bypass 回数カウント)を将来的に備えるべき。
- **CIL v0.2 時点の自己参照と派生ドキュメントの更新漏れ**:Step 14d で CIL v0.2 を読み直した際、以下の 3 件の更新漏れが発見された:(1) CI-DOC-SAD が「未着手」のまま(Step 12c で SAD v0.1 完成済)、(2) CI-DOC-CCB が「未着手」のまま(Step 10c で CCB v0.1 完成済)、(3) **CI-DOC-CIL 自己参照が v0.1 のまま**(本書 CIL 自身は v0.2 に昇格しているのに、自己参照行は v0.1 のまま)。**根本原因:**CIL v0.2 コミット(Step 12b)時に、変更理由として意識していた「CI-DOC-SRS v0.1 昇格」「CI-DOC-RMF v0.2 昇格」に作業が集中し、**台帳の網羅性チェック**(全 CI ドキュメントの最新状態が現在の成果物に一致するか)が漏れた。**対策:**CIL を更新する際は、(a) 今回の CR で直接変更する CI、(b) 自己参照(CI-DOC-CIL)、(c) 過去ステップで完成したが CIL に未反映の CI、の 3 カテゴリを必ず全走査する。Step 14d ではこれを補正し、v0.3 で 3 件を全て正しいバージョンに更新した。**お手本的価値:**台帳ドキュメントの自己参照(`CI-DOC-CIL` が CIL 自身を指す)はよく忘れられるパターン。CRR・SCMP などの自己参照台帳でも同じ注意が必要。
