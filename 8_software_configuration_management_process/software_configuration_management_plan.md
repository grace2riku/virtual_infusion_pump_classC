## 3. 構成識別(箇条 8.1)

### 3.1 構成アイテム識別手段の確立(箇条 8.1.1)

すべての構成アイテムは以下の識別情報をもつ。

- **一意の識別子(CI ID):** 種別プレフィクス + 連番(例: `CI-DOC-001`、`CI-SRC-012`、`CI-SOUP-003`)
- **名称**
- **種別:** ドキュメント / ソースコード / 試験資産 / SOUP / ツール / 環境定義 / ビルド成果物
- **バージョン:** Git コミットハッシュ(SHA-1)、リリースタグ、またはドキュメント冒頭のバージョン番号
- **保管場所:** リポジトリ内パス、または外部保管の場合は URL・参照方法

識別の実現手段:

| 手段 | 内容 |
|------|------|
| バージョン管理システム | Git(GitHub ホスティング)。<https://github.com/grace2riku/virtual_infusion_pump_classC> |
| コミット識別 | SHA-1 コミットハッシュ |
| リリースタグ規則 | `v{MAJOR}.{MINOR}.{PATCH}`(セマンティックバージョニング) |
| ベースラインタグ規則 | Inc. 完了時: `inc1-baseline` / `inc1-requirements-frozen` / `inc1-design-frozen` 等の英名タグ |
| ドキュメント識別 | ファイル冒頭に **ドキュメント ID + バージョン** を明記。ドキュメントID は種別プレフィクス(SDP/SRS/SAD/SDD/SRMP/SCMP/SSC/RMF/SPRP/SMP/CIL 等)+ 製品コード(VIP)+ 連番 |
| CI 一覧の一元管理 | [`configuration_item_list.md`](./configuration_item_list.md) にて全 CI を一覧管理 |

### 3.2 SOUP の識別(箇条 8.1.2)

各 SOUP について、以下の情報を CIL に記録する。Python 標準ライブラリおよび CPython ランタイムも SOUP として扱う。

| 項目 | 記録内容 |
|------|----------|
| SOUP ID | `SOUP-001` 等の連番 |
| 名称 | パッケージ名 |
| 製造業者/供給元 | パッケージ作者・PyPI リリーサ |
| 一意な識別子 | PyPI プロジェクト名 + リリース |
| バージョン | セマンティックバージョン(固定バージョンで依存宣言) |
| ライセンス | SPDX 識別子(例: MIT、BSD-3-Clause、Apache-2.0) |
| チェックサム | PyPI が発行する wheel ファイルの SHA-256(`uv lock` または `pip install --require-hashes` で検証) |
| 入手場所 / URL | PyPI URL、ソースリポジトリ URL |
| 入手日 | `YYYY-MM-DD` |
| 使用目的 | VIP-CTRL 内での利用箇所 |
| 安全クラス | 依存先の利用箇所に応じて A/B/C を割当て |
| 脆弱性情報源 | `pip-audit`、GitHub Advisory Database、CVE/NVD |

バージョン固定は `pyproject.toml` + ロックファイル(`uv.lock` または `requirements.txt` with hashes)で実施し、ロックファイルも CI として管理する。

### 3.3 システム構成ドキュメンテーションの識別(箇条 8.1.3)

ソフトウェアシステム構成を定義するドキュメントを識別し、構成管理下に置く。

| ドキュメント種別 | 管理方法 |
|----------------|----------|
| ソフトウェアマスタ仕様書(5.8) | リリースタグと紐付け。リリース時にスナップショットを固定 |

## 4. 変更制御(箇条 8.2)

### 4.1 変更要求の承認(箇条 8.2.1)

すべての構成アイテムへの変更は、承認された **変更要求(CR: Change Request)** に基づいて実施する。本プロジェクトでは CR を GitHub Issue(ラベル `change-request`)で管理し、対応する PR にリンクする。

| CR 区分 | 判定基準 | 承認プロセス |
|--------|---------|-------------|
| **軽微** | 誤字、コメント修正、整形、リンク修正、CI 設定の非機能変更 | 作成者(k-abe)のセルフ承認。PR + CI Pass のみ |
| **中程度** | 非安全関連機能の追加・変更、非 RCM モジュールの変更、ドキュメントの章追加 | CCB プロセス(§4.1.1)を経由。24 時間以上の検討期間を設け、自己レビューチェックリストを適用 |
| **重大** | 安全関連(RCM 実装部・クラス C 項目)、アーキテクチャ変更、要求変更、SOUP 追加/更新、リリース影響のある変更 | CCB プロセス + リスク影響解析(SRMP §7)必須。RMF 更新を伴う |

#### 4.1.1 CCB プロセス(単独開発版)

単独開発のため CCB は作成者が兼任するが、以下の規律で独立性を擬制する:

1. **Issue 起票:** 変更内容・理由・影響想定を記述
2. **インターバル:** 中程度以上は起票から 24 時間以上経過後に PR 作成開始(即興的判断を防ぐ)
3. **自己レビューチェックリスト:** PR テンプレートに埋め込み、すべての項目を Pass
4. **CI 機械検証:** 全チェック緑必須(ドキュメント lint、コード lint、試験、網羅率、脆弱性スキャン)
5. **判定記録:** PR マージコメントに承認判定と根拠を記載

#### 4.1.2 変更要求の記録項目

| 項目 | 内容 |
|------|------|
| CR ID | `CR-NNNN`(通し番号、GitHub Issue 番号を流用可) |
| 要求者・要求日 | Issue 作成者・作成日 |
| 変更対象 | 影響する CI ID(複数可) |
| 変更理由 | 問題報告 ID(PRB-NNNN)、保守要求、規格要求、改善提案等 |
| 影響範囲解析結果 | 影響する SRS / ARCH / UNIT / 試験 |
| リスクマネジメント評価 | SRMP §7.2 のチェックリスト結果、RMF 追記有無 |
| 区分 | 軽微 / 中程度 / 重大 |
| 承認者・承認日 | k-abe・PR マージ日 |
| 関連 PR | PR 番号 |

台帳は [`change_request_register.md`](./change_request_register.md) に集約する。

### 4.2 変更の実装(箇条 8.2.2)

承認後の変更実装は以下のルールに従う。

- **ブランチ戦略:** GitHub Flow。`main` から機能ブランチを切り、PR で統合
- **ブランチ名:** `feature/<短い説明>` / `fix/<問題ID>` / `docs/<対象>` 等
- **PR 必須:** `main` への直接コミットは禁止(GitHub ブランチ保護ルールで強制)
- **コミットメッセージ:** Conventional Commits 形式。CR ID を本文に含める(例: `Refs: CR-0042`)
- **レビュー:** 単独開発のためセルフレビュー + CI 機械検証
- **CI 自動実行:** ドキュメント lint(markdownlint / lychee)、コード lint(ruff / pylint / mypy / bandit)、ユニット試験(pytest)、網羅率(pytest-cov)、SOUP 脆弱性スキャン(pip-audit)

### 4.3 変更の検証(箇条 8.2.3)

変更は、その規模と安全性影響に応じた検証を経て受け入れる。

| 変更規模 | 最低限の検証 |
|---------|-------------|
| 軽微 | CI 全 Pass(lint、該当ユニット試験) |
| 中程度 | CI 全 Pass + 該当機能のユニット・結合試験 Pass + リグレッション試験 Pass |
| 重大 | 上記 + システム試験 Pass + リスク再評価(RMF 更新) + 関連ベースラインの再確認 |

### 4.4 変更のトレーサビリティ(箇条 8.2.4)

以下の連鎖を維持する。

```
[問題報告 PRB-nnnn / 保守要求 / 改善提案] → [CR-nnnn] → [Issue/PR] →
   [コミット(SHA)] → [マージコミット] → [ベースラインタグ] → [リリースタグ]
```

具体的な手段:

- Issue 記述に関連 SRS / RCM / ハザードを明記
- PR 本文に CR ID、関連 Issue、影響する CI、リスク評価結果を記載(PR テンプレートで強制)
- コミットメッセージに `Refs: CR-nnnn` を含める
- リリースノートに含まれる CR を列挙
- CI で整合性検査(CR ID と Issue の存在、PR タイトル形式)を自動化する(後続で追加)

## 5. 構成状態記録(箇条 8.3)

各ベースライン・リリース時点の構成アイテム一覧・状態を記録し、いつでも参照可能とする。

### 5.1 ベースライン設定タイミング

以下のタイミングでベースラインを凍結する。

| タイミング | タグ例 |
|-----------|-------|
| 基盤ドキュメント承認時(M0 終了) | `planning-baseline` |
| 各 Inc. の要求凍結時 | `inc{N}-requirements-frozen` |
| 各 Inc. の設計凍結時 | `inc{N}-design-frozen` |
| 各 Inc. の試験完了時 | `inc{N}-baseline` |
| 各リリース時 | `v{MAJOR}.{MINOR}.{PATCH}` |

### 5.2 状態記録の内容

各ベースラインで以下を記録する。

| 項目 | 内容 |
|------|------|
| ベースライン ID | `BL-{YYYYMMDD}-{NNN}`(例: `BL-20260418-001`) |
| ベースライン日 | `YYYY-MM-DD` |
| Git タグ | 該当タグ名 |
| Git コミット | SHA-1(完全な 40 文字) |
| 含まれる CI とバージョン | CIL の該当時点スナップショット(CIL に追記セクションとして残す) |
| 関連 CR | ベースラインに含まれる CR ID 一覧 |
| 承認者・承認日 | k-abe・日付 |
| 備考 | リリース目的・変更概要 |

GitHub Releases 機能を活用し、タグごとにリリースノートとアーカイブを作成する。

### 5.3 保管

- **Git リポジトリ:** GitHub 上で永続保管。タグ・コミットは書き換え不可(force push は main で禁止)
- **ローカルバックアップ:** 定期的なミラーリング(必要に応じて外部ストレージへ)
- **保管期間:** 実製品では医療機器の寿命 + 規定期間のアーカイブが必要。本プロジェクトは仮想のため学習アーカイブとして保持(実製品展開時は保管計画を SMP に追記)
- **削除制限:** ベースラインタグ・リリースタグは削除禁止(ブランチ保護ルール + GitHub の保護されたタグ設定で強制)

## 8. ブランチ保護ルール(GitHub)

`main` ブランチに対して以下を設定する(セットアップ予定):

- [ ] PR 必須(直接 push 禁止)
- [ ] CI チェック Pass 必須
- [ ] PR を最新の main に対して更新必須(stale な PR は再同期が必要)
- [ ] 管理者にもルール適用
- [ ] force push 禁止
- [ ] ブランチ削除禁止
- [ ] タグ保護:リリースタグ・ベースラインタグは削除/上書き禁止
