## 1. 目的と適用範囲

本規程は、IEC 62304 箇条 8.2(変更制御)および SCMP-VIP-001 §4.1 に基づき、変更管理委員会(CCB: Change Control Board)の構成・運営・意思決定手順を定義する。本規程は Virtual Infusion Pump Control Software(VIP-CTRL)のソースコード・ドキュメント・SOUP・ツール・試験資産・CI 設定に対するすべての変更要求(CR)に適用する。

本規程は SCMP §4.1.1 の「単独開発向け CCB プロセス」を具体化し、独立した文書として監査時のトレーサビリティを確保することを目的とする。

## 3. CCB の構成

### 3.1 標準モデル(実製品運用想定)

| 役割 | 所属 | 責任 |
|------|------|------|
| 議長 | 品質保証部門 | 会議運営、最終承認、区分判定の裁定 |
| 書記 | 品質保証部門 | 議事録、CRR 更新 |
| 開発責任者 | ソフトウェア開発部門 | 技術的影響評価、実装可能性判断 |
| 試験責任者 | 検証部門 | 検証影響評価、回帰試験範囲決定 |
| RA 責任者 | 規制対応部門 | 規制影響・報告義務評価 |
| リスクマネジメント責任者 | 品質保証部門 | 安全性・リスク評価、RMF 更新判断 |

### 3.2 招集メンバ(必要に応じて)

- サービス部門
- カスタマーサポート部門
- マーケティング部門
- 外部専門家・コンサルタント

### 3.3 定足数(標準モデル)

- **通常 CR:** 常任メンバの過半数、かつ RA 責任者または RM 責任者のいずれかを含む
- **安全関連・回収関連 CR:** RA 責任者 + RM 責任者 + 品質保証責任者の全員参加

### 3.4 本プロジェクトの兼任運用(単独開発版)

本プロジェクトでは k-abe が以下の全役割を兼任する:

| 標準モデルでの役割 | 本プロジェクトでの担当 | 独立性の代替手段 |
|-----------------|---------------------|----------------|
| 議長・書記 | k-abe | GitHub Issue + PR で記録(機械的監査証跡) |
| 開発責任者 | k-abe | コードは単独記述 |
| 試験責任者 | k-abe | CI による機械的試験 Pass 必須 |
| RA 責任者 | k-abe | SMP §4.7 により本プロジェクトでは規制通知は運用なし |
| リスクマネジメント責任者 | k-abe | RMF 更新必須、SRMP §7.4 影響解析チェックリスト遵守 |

## 4. CR の起票と受付

### 4.1 起票可能な者

- 開発者・試験者(= k-abe、本プロジェクトでは全員)
- (実製品展開時)カスタマーサポート・サービス、品質保証担当者、RA 担当者、外部要求を受領した担当部門

### 4.2 起票手段

本プロジェクトでは **GitHub Issue** を CR の起票手段とする。ラベル `change-request` を付与する。Issue 番号が CR 番号となる(`CR-NNNN` = Issue #NNNN)。

### 4.3 起票時の記載項目

SCMP §4.1.2 および `.github/ISSUE_TEMPLATE/change_request.md` に従う。主要項目:

- CR ID(Issue 番号)
- 起票者(GitHub handle)、起票日
- 変更対象(CI ID、ファイルパス)
- 変更理由(関連 PRB-NNNN、保守要求、規格要求、SOUP 更新、改善提案)
- 変更区分(軽微 / 中程度 / 重大)
- 影響範囲解析
- SRMP §7.2 リスク影響解析チェックリスト
- RMF 更新要否
- 実装見積もり・完了目標マイルストーン
- 検証計画
- リリース方針

## 5. CR の分類と承認フロー

### 5.1 区分定義(SCMP §4.1 との整合)

| 区分 | 定義 | 承認フロー(標準モデル) | 本プロジェクトでの運用 |
|------|------|----------------------|---------------------|
| 軽微 | 誤字、コメント、整形、リンク修正、CI 設定の非機能変更 | 開発責任者のみ | セルフ承認、即時 PR 可 |
| 中程度 | 非安全関連機能の追加・変更、非 RCM モジュールの変更、ドキュメント章追加 | CCB 簡易審議 | セルフ承認 + **24 時間インターバル** + 自己レビューチェックリスト + CI 全 Pass |
| 重大 | 安全関連(RCM 実装部・クラス C 項目)、アーキテクチャ変更、要求変更、SOUP 追加/更新、リリース影響 | CCB 正式審議 | セルフ承認 + 24 時間インターバル + 自己レビューチェックリスト + **RMF 更新** + **SRMP §7.4 影響解析** + CI 全 Pass |

### 5.2 重大判定の基準

以下のいずれかに該当する場合は **重大** と判定する(CR 区分判定用チェックリスト):

- [ ] リスクコントロール手段(RCM-xxx)を実装する CI への変更
- [ ] ソフトウェア安全クラス(クラス C 項目)に影響し得る変更
- [ ] 分離設計(IEC 62304 §5.3.5)に影響する変更
- [ ] 外部インタフェース(仮想ハードウェアモデル I/F 含む)の変更
- [ ] SOUP の追加・削除・バージョン変更(MAJOR 更新を含む)
- [ ] 規格要求・規制要求への対応を伴う変更
- [ ] リリース影響のある変更(SemVer の MINOR 以上に該当)
- [ ] (実製品展開時)規制当局への通知義務が生じ得る変更、市販済み機器へのフィールド対応

### 5.4 24 時間インターバルの意義と遵守方法

**目的:** 起票直後の即興的判断を避け、頭を冷やしての再確認時間を確保する。CCB 複数名審議の「独立視点による再検討」を時間差で擬制する。

**遵守方法:**

- Issue 起票時刻と PR 作成時刻の差分が 24 時間以上であること(GitHub 上で機械的に追跡可能)
- インターバル中に別タスク・休憩を挟み、同一セッション内での連続作業を避ける
- インターバル終了時に自己レビューチェックリストを改めて最初から実施する

**例外:**

- 軽微区分は対象外
- 重大区分でも、以下に該当する場合はインターバル短縮可(ただし理由を PR に明記):
  - 安全性影響のある重大不具合のホットフィックス(SMP §5.4 参照)
  - 規制当局指摘への緊急対応(本プロジェクトでは運用なし)

## 6. CCB 会議の運営(標準モデル)

### 6.1 定例会(標準モデル)

- 頻度: 週次 または 隔週
- 時刻: 組織ルールで決定
- 形式: 対面・ハイブリッド・リモートのいずれか

### 6.2 臨時会(標準モデル)

以下のトリガで議長が臨時招集する。

- 安全関連 CR の発生
- 市販後重大不具合の発覚
- 規制当局からの照会・指示
- リリース直前の重大判断

### 6.3 本プロジェクトでの「会議」代替

単独開発のため物理的会議は行わない。代替として:

- **審議の代替:** GitHub Issue 上で区分判定チェックリスト・影響解析・リスク評価を記入することをもって「審議」とする
- **議事録の代替:** Issue 本文・コメント・PR レビューコメント・PR マージコメントを議事録として扱う
- **議事進行の代替:** Issue テンプレートの項目順に記入することで議事進行を擬制
- **定足数の代替:** CI 全 Pass と 24h インターバル完了をもって定足数成立と見なす

## 7. 審議の観点

各 CR は以下の観点で審議する(Issue 本文に記入する必須項目)。

- [ ] 変更の必要性・妥当性(目的と期待効果)
- [ ] 技術的影響範囲(依存する CI、回帰試験範囲)
- [ ] 安全性への影響(RMF 更新の要否、関連ハザード)
- [ ] 規制への影響(本プロジェクトでは主に SMP §4.7 のリリースノート記載事項の判定)
- [ ] スケジュール・マイルストーン影響
- [ ] ロールバック計画(変更失敗時の復旧手順)
- [ ] 関連する既存 PRB-NNNN との整合

## 8. 決定の記録

CCB の決定は以下を記録する。本プロジェクトでは GitHub の PR マージコメントおよび CRR-VIP-001(変更要求台帳)に記録する。

| 項目 | 記載内容 |
|------|----------|
| 決定日 | PR マージ日(`YYYY-MM-DD`) |
| CR ID | `CR-NNNN`(Issue 番号) |
| 決定種別 | 承認 / 条件付き承認 / 差戻し / 却下 / 保留 |
| 条件(条件付き承認時) | 例:「特定の試験 Pass を別 PR で先行」 |
| 出席者(本プロジェクト) | k-abe(セルフ、24h インターバル遵守の旨を記載) |
| 審議サマリ | Issue 上のチェックリスト結果 |
| 影響解析結果 | SRMP §7.2 チェックリスト結果 |
| RMF 更新 | 実施 / 不要(実施時はエントリ ID) |
| 次回確認事項 | 後続フォローアップが必要な場合(例: 試験報告の追補) |
| マージコミット SHA | SHA-1 |

## 9. CCB の監査と継続的改善

- **四半期ごと**(本プロジェクトでは各 Inc. 完了時)に過去の CR を振り返り、以下を SPRP §8 傾向分析と併せて評価:
  - CR の 24h インターバル遵守率
  - CR 区分判定の妥当性(後から区分を変更した件数)
  - RMF 更新漏れの有無
  - CI 失敗に起因する CR の件数(プロセス改善シグナル)

- 評価結果に基づき本規程を改訂する。改訂は本規程自体を対象とした「重大」区分の CR として扱う(メタ運用)。
