## 3. ソフトウェア開発プロセスの概要(箇条 5.1.1)

### 3.1 ライフサイクルモデル

採用するライフサイクルモデル: **V字モデル(インクリメンタル方式)**

- 各機能インクリメントに対して独立した V字サイクル(要求分析→アーキテクチャ設計→詳細設計→実装→ユニット試験→結合試験→システム試験)を一周する。
- インクリメントごとに SRS・SDD 等の関連ドキュメントを追補し、ベースラインを更新する。
- 全インクリメント完了後、通しでのシステム試験と最終ベースラインの凍結(5.8 リリース)を行う。

採用の根拠:

1. V字モデルは IEC 62304 の各条項(5.2→5.3→5.4→5.5→5.6→5.7)と工程が 1 対 1 で対応し、トレーサビリティを視覚的に維持できる。
2. 単一V字で全機能を一気通貫に開発するのは規模的に非現実的なため、機能単位のインクリメンタル分割を採用する。
3. AAMI TIR45 で許容される開発形態の範囲内で、規律(成果物の完全性・バージョン管理・トレース)を維持しつつ学習効果を最大化できる。

### 3.2 インクリメント計画

| Inc. | 機能範囲 | 代表的な要求項目 |
|------|---------|----------------|
| Inc.1 | 流量制御コア | 流量・投与量・投与時間の設定保持、定常注入、停止、状態管理、仮想ポンプモデル |
| Inc.2 | アラーム管理 | 閉塞検知、気泡検知、薬液切れ検知、バッテリ低下、アラーム通知・確認・消音 |
| Inc.3 | 用量計算・ボーラス | 単位変換、体重換算、ボーラス投与、段階的投与スケジュール |
| Inc.4 | ロギング・UI | 注入履歴・アラーム履歴の記録と表示、操作 UI、設定入力の確定ロジック |

各インクリメントは V字を一周し、終了時点で該当機能のシステム試験までを完了する。後続インクリメントは前インクリメントの成果物を前提としてリグレッション試験も含めて実施する。

## 6. ソフトウェア開発標準・方法・ツール計画(箇条 5.1.4 ― クラス B, C)

### 6.1 コーディング規約

- **使用言語:** Python 3.12
- **適用規約:** PEP 8(Python 標準コーディング規約)、プロジェクト独自の命名規約(アーキテクチャ設計書で詳細化)
- **型チェック:** mypy(strict モード)による静的型検査を必須とする
- **静的解析ツール:** pylint、ruff、mypy、bandit(セキュリティ)
- **セキュアコーディング:** CWE 参照、bandit による検出、入力検証の徹底
- **言語選択の根拠:** 本プロジェクトは PC 単体で動作する仮想シミュレータを対象とし、実機マイコン実装は行わない。学習目的・記述効率・シミュレーション表現力を重視し Python を採用する。MISRA-C 等の C 系規約は適用できないため、代替として型ヒント + 静的解析 + 徹底したユニット試験で品質を担保する。実機展開する場合は言語再選定を行う。

### 6.3 採用方法論

- **テスト駆動開発(TDD)** を原則として採用する。ユニットはテストファーストで作成する。
- **ペアレビューは単独開発のためチェックリスト自己レビューに代替する**。重要モジュールはレビューまでに 24 時間以上のインターバルを設ける。
- **リファクタリング** は試験がグリーンな状態でのみ行い、試験追加を伴う。

## 7. ソフトウェア統合及び統合試験の計画(箇条 5.1.5)

- **統合戦略:** ボトムアップ統合
  - Step 1: 仮想ハードウェアモデル(ポンプ駆動、センサー)の単体確認
  - Step 2: 制御コア(流量制御、状態管理)と仮想ハードウェアの結合
  - Step 3: アラーム管理サブシステムの統合
  - Step 4: 用量計算・ボーラスロジックの統合
  - Step 5: UI・ロギングの統合
- **統合環境:** PC(macOS / Linux)上の Python 実行環境。外部ハードウェアは不要。仮想ハードウェアモデルは結合試験時にシナリオ入力(閉塞発生、気泡発生、薬液切れ等)を注入可能なスタブを提供する。
- 各インクリメントの V字右腕で該当範囲の結合試験を実施し、以降のインクリメント完了時にリグレッション結合試験を実施する。

| 仕様書類 | 本書含む全 Markdown ドキュメント | Git で版管理、リリースタグでベースライン化 |

## 13. 検証前の構成アイテムの制御(箇条 5.1.11)

- コードおよびドキュメントは、検証(レビュー・試験)に入る **前** に構成管理下に置く。
- 具体的実施方法:
  - ローカルで作業中の変更は、Pull Request 作成時点でトピックブランチへ push し、以降のレビュー・CI 試験はそのブランチ上で実施する。
  - PR マージ前に GitHub Actions による lint・ドキュメント検証・ユニット試験が全 Pass することを必須とする(CI ブロック)。
  - マージ後の main ブランチは検証済みベースラインとみなし、直接コミットは禁止(ブランチ保護ルールを設定する)。
- 単独開発のため PR レビューは自己レビューとなるが、CI による機械的検証を独立性の代替手段として位置付ける。

## 14. 共通ソフトウェア欠陥の識別と回避(箇条 5.1.12 ― クラス B, C)

選択した開発言語(Python)および採用技術(マルチスレッド、イベント駆動制御、数値計算)に固有の、共通的に知られるソフトウェア欠陥類型を識別し、回避策を定義する。

| 欠陥類型 | 該当技術 | 回避策・検出手段 |
|---------|---------|---------------|
| 型の取り違え・暗黙型変換ミス | 動的型付け(Python) | mypy strict、pydantic による実行時型検証、型ヒント必須化 |
| 整数オーバーフロー・浮動小数点誤差 | 用量計算 | 安全な算術(Decimal)、境界値試験、hypothesis によるプロパティ試験 |
| ゼロ除算・範囲外入力 | 用量計算・時間計算 | 入力検証、防御的プログラミング、ユニット試験の境界値網羅 |
| 未定義変数参照 | Python | pylint、ruff、pyright |
| 並行処理不具合(競合・デッドロック) | アラーム管理・タイマ | 設計レビュー、ロック順序規約、pytest によるシナリオ試験 |
| 資源リーク(ファイル・接続) | ロギング | with 構文の徹底、静的解析 |
| 例外処理漏れ・例外の握り潰し | 全般 | ruff による except の監視、特定例外のみ捕捉する規約、ロギング必須化 |
| 入力検証漏れ(インジェクション等) | UI 入力 | bandit、pydantic、入力境界での検証一元化 |
| SOUP 起因の既知脆弱性 | 外部ライブラリ | 依存脆弱性スキャン(pip-audit 等)、SOUP 台帳で追跡 |
| 状態遷移の不整合(有り得ない状態への遷移) | 制御状態機械 | 状態機械の形式化、不変条件のアサート、到達可能性確認 |
| 浮動小数比較による誤判定 | 数値計算 | Decimal 使用、許容誤差付き比較ヘルパ関数 |

参照: CWE、CERT Python Secure Coding、OWASP、FDA Guidance "Content of Premarket Submissions for Software Contained in Medical Devices" 等。
