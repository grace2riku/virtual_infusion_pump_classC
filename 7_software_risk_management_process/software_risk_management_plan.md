## 1. 目的と適用範囲

本書は、IEC 62304 箇条 7(ソフトウェアリスクマネジメントプロセス)および ISO 14971:2019 に基づき、Virtual Infusion Pump Control Software(VIP-CTRL) のリスクマネジメント活動を **計画** する。本書は「プロセスの計画」であり、具体的なハザード・原因・RCM・検証記録の実体は **リスクマネジメントファイル(RMF)** にて管理する。本書とRMFの関係は以下のとおり:

- **SRMP(本書):** プロセスの方針・方法・判断基準・役割・タイミングを定義する。
- **RMF:** 当該プロセスに従って識別された個々のハザード、原因、RCM、検証記録、残留リスク評価を時系列で記録する生きた文書。
- **SSC:** ソフトウェア安全クラスの決定根拠を記録した静的文書。

本プロセスの出力は、本プロジェクトにおいては RMF に統合する(実製品ではシステムレベル RMF に統合する想定)。

## 3. リスクマネジメントプロセスの方針

### 3.1 活動の全体フロー

```
[ハザード識別] → [潜在原因識別] → [事象シーケンス文書化] → [RCM 定義]
   → [RCM 実装(SRS→設計→コード)] → [RCM 検証(試験・解析)]
     → [残留リスク評価] → [受容可否判定] → [RMF 更新]
```

### 3.2 役割と責任

| 役割 | 責任 |
|------|------|
| リスクマネージャ(本プロジェクトでは作成者が兼任) | 本計画の策定・更新、RMF の一貫性維持、残留リスク受容の判断 |
| 開発者(本プロジェクトでは作成者が兼任) | RCM の実装、試験、トレーサビリティ確保 |
| 検証者(本プロジェクトでは作成者が兼任) | 独立性担保のための時間間隔とチェックリストに基づく独立検証、CI による機械的検証 |

> 単独開発のため役割は一人で兼任するが、活動ごとに作業日を分ける / 自己レビューチェックリストを用いる / CI による機械的検証を独立性の代替手段とする。

### 3.3 活動のタイミングとトリガ

| タイミング | 活動 |

### 3.4 リスク受容基準

ISO 14971 に従い、ALARP(As Low As Reasonably Practicable)の原則で判断する。クラス C ソフトウェアの本プロジェクトに適用する基準:

| 残留リスクレベル | 判定 |
|-----------------|------|
| 死亡/重傷につながるハザードで、RCM 未実装または検証で不合格 | **受容不可**(リリース不可) |
| 死亡/重傷につながるハザードで、RCM が少なくとも 1 段(多くは冗長に 2 段以上)実装・検証合格 | 条件付き受容(RMF に根拠・残留リスク評価を明記) |
| 軽度傷害/健康被害なしで、通常の防御的プログラミング・試験網羅で対処 | 受容可 |

RCM の優先順位は ISO 14971 の階層に従う:

1. **設計による本質安全**(ハザード原因の排除・構造的不能化)
2. **保護手段**(防御的チェック、冗長化、フェイルセーフ動作、アラーム)
3. **警告・使用情報**(操作者への通知、取扱説明書記載)

本プロジェクトは仮想機器であり、ハードウェア外部 RCM は存在しないと扱うため、すべての RCM はソフトウェアで実装する。

## 4. ハザード状況に寄与するソフトウェアの解析(箇条 7.1)

### 4.1 ハザード状況に寄与するソフトウェア項目の識別(箇条 7.1.1)

システムリスク解析の出発点として、SSC-VIP-001 で識別された HZ-001〜HZ-008 を初期ハザードとして採用する。各ハザードに寄与するソフトウェア項目はアーキテクチャ設計(SAD)完了時に確定し、RMF に記録する。初期ハザード一覧と想定される寄与項目カテゴリを以下に示す。

| ハザード ID | ハザード状況 | 寄与が想定されるソフトウェア項目(カテゴリ) | 安全クラス |
|-----------|-----------|-----------------------------------------|----------|
| HZ-001 | 過量投与 | 流量制御コア、用量計算、状態管理 | C |
| HZ-002 | 過少投与・意図しない注入停止 | 流量制御コア、状態管理、タイマ制御 | C |
| HZ-003 | 誤薬剤プロファイル適用 | 薬剤ライブラリ管理、設定確定ロジック、UI 入力検証 | C |
| HZ-004 | 閉塞・気泡・薬液切れの検知失敗 | アラーム管理、センサー入力処理、仮想ハードウェア I/F | C |
| HZ-005 | アラーム失敗 | アラーム管理、スケジューラ、通知 I/F | C |
| HZ-006 | 流量・ボーラス投与量の計算誤り | 用量計算、単位変換、ボーラス制御 | C |
| HZ-007 | 電源喪失・再起動時の不定動作 | 永続化、起動時状態復元、状態管理 | C |
| HZ-008 | 意図しない操作の確定 | UI 入力処理、確認ダイアログ、デバウンス制御 | C |

実体(ARCH-xxx / UNIT-xxx の具体 ID との紐付け)は RMF で管理する。

### 4.2 寄与する潜在原因の識別(箇条 7.1.2)

各ソフトウェア項目について、以下のカテゴリ別に潜在原因を系統的に識別する。カテゴリはチェックリストとして運用する。

| 潜在原因カテゴリ | 具体例(輸液ポンプ文脈) | 関連する SDP §14 欠陥類型 |
|-----------------|-----------------------|---------------------------|
| ソフトウェア機能の不正 | 要求仕様誤り、ロジック誤り、状態遷移誤り | 状態遷移の不整合 |
| 入出力の失敗 | センサー値の取り違え、単位混在、表示誤り | 型の取り違え、入力検証漏れ |
| タイミング | デッドライン超過、競合状態、アラーム遅延 | 並行処理不具合 |
| SOUP 起因 | 既知不具合、脆弱性、非互換更新 | SOUP 起因脆弱性 |
| 資源枯渇 | メモリ枯渇、ファイル記述子枯渇、スレッド枯渇 | 資源リーク |
| 数値計算 | 桁落ち、オーバーフロー、ゼロ除算、浮動小数誤差 | オーバーフロー、浮動小数比較 |
| 例外処理 | 例外未捕捉、握り潰し、ログ欠落 | 例外処理漏れ |
| 永続化 | データ破損、起動時復元誤り、競合書き込み | 資源リーク |
| 操作系 | 誤タッチ確定、デバウンス不良、競合入力 | 入力検証漏れ |

### 4.3 SOUP 公開異常リストの評価(箇条 7.1.3)

本プロジェクトで使用する SOUP は、以下の手順で既知不具合・脆弱性を評価する。

- **識別:** 採用するすべての Python パッケージ(直接依存・推移的依存を含む)を CIL に登録する。Python 標準ライブラリ・CPython ランタイムも SOUP として扱う。
- **情報源:**
  - Python 公式 Security Announcements(<https://www.python.org/security/>)
  - 各パッケージのリリースノート・Issue Tracker
  - CVE / NVD(<https://nvd.nist.gov/>)
  - GitHub Advisory Database(<https://github.com/advisories>)
  - `pip-audit` による自動脆弱性スキャン(CI 組込予定)
- **頻度:** 採用時・バージョン更新時・リリース前、および少なくとも各インクリメント境界で実施。
- **評価観点:** ソフトウェア機能として使用している箇所を特定し、当該不具合/脆弱性が HZ-001〜HZ-008 のいずれかに寄与するかを判定する。
- **記録:** 評価結果は CIL の SOUP 節および RMF に記録する。

現時点で具体の SOUP は未確定のため、エントリはアーキテクチャ設計・実装開始時に追加する。

### 4.4 潜在原因の文書化(箇条 7.1.4)

識別された潜在原因は以下の項目で RMF に記録する。

- 原因 ID(例: `CAUSE-HZ001-001`)
- 対応するハザード状況 ID(例: `HZ-001`)
- 寄与するソフトウェア項目(ARCH-xxx / UNIT-xxx)
- 潜在原因カテゴリ(§4.2 のカテゴリ)
- 危害の重大度評価(ISO 14971 基準)
- 発生確率評価(可能な範囲で)
- 初期リスクレベル

### 4.5 事象シーケンスの文書化(箇条 7.1.5)

潜在原因からハザード状況・危害に至る事象シーケンスを明示的に文書化する。形式は因果連鎖とする。

**例(輸液ポンプ文脈):**

```
原因: 用量計算における単位変換の丸め誤差 (CAUSE-HZ006-003)
 ↓
中間状態: 内部流量指令値が仕様の10倍
 ↓
ハザード状況: 過量投与 (HZ-001)
 ↓
危害: 患者への薬剤過量投与 → 低血糖昏睡/循環虚脱/死亡
```

```
原因: アラームタスクのデッドロック (CAUSE-HZ005-001)
 ↓
中間状態: 閉塞発生もアラーム発報しない
 ↓
ハザード状況: 閉塞検知の実効失敗 (HZ-004 + HZ-005)
 ↓
危害: 薬剤途絶の放置 → 致死的転帰
```

すべての事象シーケンスは RMF に個別に記録する。

## 5. リスクコントロール手段(箇条 7.2)

### 5.1 リスクコントロール手段の定義(箇条 7.2.1)

各潜在原因に対して、ISO 14971 の階層(§3.4)に従って RCM を定義する。

| RCM 種別 | 想定される手段(輸液ポンプ文脈の例) |
|---------|----------------------------------|
| 設計による本質安全 | 状態機械による不能状態の構造的排除、型システムによる単位混在の静的防止、Decimal 使用による浮動小数誤差の構造的排除 |
| 保護手段 | 流量指令値の範囲チェック、冗長計算(異なるアルゴリズムでの二重計算と突合)、ウォッチドッグ、自己診断、フェイルセーフ停止 |
| 警告・使用情報 | アラーム音・視覚通知、ログ出力、取扱説明書記載 |

RCM は `RCM-xxx` の形式で識別し、以下の属性で定義する:

- RCM ID
- 対象原因 ID(1 対多可)
- 手段の種別(上記 3 階層)
- 実装先(ARCH-xxx または UNIT-xxx)
- 検証方法(予定)
- 検証試験 ID(予定)

### 5.2 ソフトウェアで実装されるリスクコントロール手段(箇条 7.2.2)

ソフトウェアで実装する RCM は以下の追跡を徹底する。

1. **SRS への要求事項としての組込み:** RCM の機能要求を SRS に `SRS-xxx` として記述し、本文中に `RCM-xxx` を参照明記する。
2. **アーキテクチャ設計での配置:** SAD において RCM を実装するソフトウェア項目を特定する。クラス C 分離設計(箇条 5.3.5)の対象にもなる。
3. **詳細設計での実装:** SDD においてアルゴリズム・インタフェース・不変条件を記述する。
4. **コードでのタグ付け:** 該当関数/モジュールのドキュメンテーションに `# RCM-xxx` コメントを付与し、grep 可能にする。
5. **試験での検証:** ユニット試験・結合試験・システム試験で RCM の動作を確認する。試験ケースに `RCM-xxx` をタグ付けする。

## 6. リスクコントロール手段の検証(箇条 7.3)

### 6.1 リスクコントロール手段の検証(箇条 7.3.1)

各 RCM は以下のいずれかの方法で検証し、記録を残す。

| RCM 種別 | 推奨される検証方法 |
|---------|-------------------|
| 設計による本質安全 | 設計レビュー、静的解析、型チェック、(必要に応じて)形式手法 |
| 保護手段(防御的ロジック) | ユニット試験(境界値・異常系・hypothesis によるプロパティ試験) |
| 保護手段(冗長化・フェイルセーフ) | 結合試験(故障注入シナリオ) |
| 保護手段(ウォッチドッグ等) | 結合試験/システム試験(遅延注入・故障注入) |
| 警告(アラーム) | システム試験(異常シナリオでの発報確認) |

記録は該当試験報告書(UT-xxx / IT-xxx / ST-xxx)に残し、RMF からトレーサブルにする。

### 6.2 新たな事象シーケンスの文書化(箇条 7.3.2)

RCM 実装自体が新たなハザード状況を生む可能性を以下の観点で評価し、該当あれば §4.5 の事象シーケンスに追加する。

- [ ] RCM が動作しない場合の挙動(安全側/危険側)
- [ ] RCM 誤動作(誤検出)による想定外停止・警告過多
- [ ] RCM 実装によるリソース消費増で別リソース枯渇を誘発
- [ ] RCM 実装が別の RCM の前提を崩す(副作用)

評価は各 RCM 実装完了時および変更発生時に実施する。

### 6.3 トレーサビリティの文書化(箇条 7.3.3)

以下のトレーサビリティを RMF および各ドキュメントのトレーサビリティマトリクスで維持する。

```
[ハザード HZ-xxx] ←→ [潜在原因 CAUSE-xxx] ←→ [RCM-xxx]
                                                 ↓
                                         [SRS-xxx] ←→ [ARCH-xxx / UNIT-xxx]
                                                                 ↓
                                                         [UT-xxx / IT-xxx / ST-xxx]
```

ID 体系は SDP §6 の命名規約に準じ、RMF で一元管理する。

## 7. ソフトウェア変更のリスクマネジメント(箇条 7.4)

### 7.1 変更がソフトウェアに与える影響の解析(箇条 7.4.1)

すべてのソフトウェア変更(要求変更、設計変更、不具合修正、SOUP 更新を含む)について、以下を解析する。

- 新たに発生し得るハザード状況
- 既存ハザード状況の重大度・発生確率の変化
- 新たな潜在原因の出現

変更は Pull Request 単位で管理し、PR 記述欄に影響解析結果を記載する。クラス C 項目に影響する変更は、影響解析結果を RMF に追記する。

### 7.2 既存 RCM への影響の解析(箇条 7.4.2)

以下のチェックリストで評価する。

- [ ] 変更範囲に RCM を実装するユニット(`# RCM-xxx` タグで検出可能)が含まれるか
- [ ] RCM が依存するインタフェース・データ構造・定数に変更があるか
- [ ] RCM の前提(例: センサー値のレンジ、タイマ精度)が変わるか
- [ ] 関連する検証記録(UT-xxx / IT-xxx / ST-xxx)は再実施が必要か
- [ ] 変更により新規 SOUP が追加・更新されるか(追加時は §4.3 を実施)

### 7.3 解析結果に基づくリスクマネジメント活動(箇条 7.4.3)

解析結果に応じて、下記の活動を選択的に再実施する。記録は RMF に残す。

| 影響の程度 | 必要活動 |
|-----------|---------|
| 新規ハザード出現 | ハザード識別 → 原因識別 → RCM 定義 → 実装 → 検証(全工程再実施) |
| 既存 RCM の論理変更 | 当該 RCM の再設計・再検証 |
| 既存 RCM の依存変更のみ(論理不変) | 当該 RCM の再検証(試験再実施) |
| RCM 非関連部の変更 | リグレッション試験範囲の判定のみ |
| SOUP 更新 | §4.3 の SOUP 評価を再実施 |
