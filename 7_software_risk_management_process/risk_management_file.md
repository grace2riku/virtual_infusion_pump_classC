## 3. 意図する使用及び安全に関連する特性(ISO 14971 箇条 5.2)

### 3.1 意図する使用

| 項目 | 記述 |
|------|------|
| 対象疾患・適応 | 静脈内投与が必要な治療全般(持続投与薬剤、電解質補充、栄養補給、昇圧剤、抗不整脈薬、鎮静剤、麻薬性鎮痛剤、インスリン等) |
| 対象ユーザ | 医療従事者(医師・看護師・薬剤師)。操作者は取扱説明書を理解した訓練済み有資格者を想定 |
| 使用環境 | 医療機関内(病棟、ICU、手術室、救急外来)。在宅利用は本版では対象外 |
| 操作頻度 | 連続使用(数時間〜複数日)。1 患者に対し 1 ポンプ以上の同時運用あり |
| 接触対象 | 薬液が患者の静脈へ直接投与される(輸液ライン経由) |
| 患者集団 | 成人・小児・新生児(体重換算投与の対応が必要) |

> 本プロジェクトは **PC 単体のシミュレータ** だが、意図する使用は実機相当を想定して記述する(安全クラス C 判定の前提)。実機展開時は本節を再検証する。

### 3.2 安全に関連する特性(ISO 14971 附属書 C)

| 項目 | 記述 |
|------|------|
| エネルギーを与えるか | 直接のエネルギー供給は無い(モータ駆動による機械的流量制御のみ) |
| 物質を投与するか | **はい** — 薬剤・栄養剤・輸液を患者に投与する(主要リスク源) |
| 生体由来物質との接触 | 輸液ラインを介して血液と間接接触の可能性(ライン自体は使い捨て滅菌) |
| 滅菌の要否 | 輸液ラインは滅菌(ディスポーザブル)、ポンプ本体は清拭消毒 |
| 継続使用時間 | 連続(時間〜日単位)。電源喪失・バッテリ切れ時の復旧動作が重要 |
| 操作ミス可能性 | **高い** — 複数薬剤・複数設定値(流量・投与量・投与時間)、単位混在(mL/h、mg/kg/min 等)、緊急時の焦りによる誤操作 |
| ソフトウェア制御 | **あり(クラス C)** — 制御アルゴリズム・アラーム・UI・計算ロジックがすべてソフトウェアで実現 |
| 外部インタフェース | 本版は単独機器。将来的に電子カルテ連携、ナースコール連携等の拡張時は再評価要 |
| 測定機能 | 積算投与量、残量、経過時間、アラーム履歴の記録・表示 |
| 一回使用/再使用 | 輸液ラインは一回使用、ポンプ本体は多数患者に再使用 |

## 4. ハザード・ハザード状況・危害の識別(ISO 14971 箇条 5.4)

### 4.1 ハザード一覧

SSC-VIP-001 で識別した 8 件を初期ハザードとして登録する。以降、SRS/SAD/実装/試験/市販後情報の各段階で追加・更新する。

| ハザード ID | ISO 14971 分類 | 記述 | 想定される危害 | ソフトウェア寄与 | 発生しうる機序(概要) |
|-----------|----------------|------|------------|--------------|-------------------|
| HZ-001 | 機能的(投与量) | 過量投与 — 設定値を超える流量・積算量での注入 | 薬剤過量による循環虚脱・中毒・死亡(麻薬・インスリン・高濃度カリウム・昇圧剤等) | **有** | 流量制御ロジック誤り、計算誤り、UI 入力誤確定、状態遷移誤り |
| HZ-002 | 機能的(投与量) | 過少投与・意図しない注入停止 | 生命維持薬剤の途絶による急性増悪・死亡(昇圧剤、抗不整脈薬、インスリン等) | **有** | 停止タイマ誤動作、ウォッチドッグ誤作動、再起動時復元失敗 |
| HZ-003 | 情報的・機能的 | 誤薬剤プロファイル適用(薬剤ライブラリ取り違え) | 薬剤に対する不適切な流量・単位換算による重傷・死亡 | **有** | 薬剤選択ロジック誤り、薬剤 DB 破損、UI 入力確定誤り |
| HZ-004 | 機能的(モニタリング) | 閉塞・気泡・薬液切れの検知失敗 | 薬剤途絶の放置、空気塞栓による脳梗塞・心停止 | **有** | センサー入力処理誤り、閾値設定誤り、検知アルゴリズム不備 |
| HZ-005 | 情報的(警告) | アラーム失敗(鳴らない・無効化・誤消音) | 異常状態の放置により上記 HZ-001〜HZ-004 の危害が顕在化 | **有** | アラームタスク DL、優先度設定誤り、通知 I/F 不具合 |
| HZ-006 | 機能的(計算) | 流量・ボーラス投与量の計算誤り(単位換算・丸め・オーバーフロー) | 想定の 10 倍・100 倍投与等による重篤な中毒・死亡 | **有** | 整数オーバーフロー、浮動小数誤差、単位変換誤り、桁落ち |
| HZ-007 | 機能的(状態管理) | 電源喪失・再起動時の不定動作(停止済み注入の誤再開、設定喪失) | 意図しない二重投与・投与中断 | **有** | 永続化破損、起動時復元誤り、再起動フロー未検証パス |
| HZ-008 | 情報的(操作) | 意図しない操作の確定(デバウンス不良、誤タッチの確定) | 意図しない用量・速度で投与される | **有** | 入力検証漏れ、確認ダイアログ不在、タッチ連続入力の誤処理 |

> ソフトウェア寄与ありのハザードは SRMP-VIP-001 §4.1 の「ハザード寄与解析」と 1 対 1 に対応する。ID が一致するように維持する。

### 4.2 事象シーケンス(代表例)

各ハザードに対する代表的な事象シーケンスを以下に示す。完全な列挙は SRS/SAD 完成時に追補する。事象 ID は `EV-HZxxx-nnn` 形式で管理する。

#### EV-HZ001-001(HZ-001 過量投与)

```
[原因] 用量計算における単位変換の丸め誤差 (CAUSE-HZ006-003 経由)
  ↓
[中間状態] 内部流量指令値が設定値の 10 倍に計算される
  ↓
[ハザード状況] 設定値を超える流量での注入 (HZ-001)
  ↓
[危害] 薬剤過量投与 → 低血糖昏睡/循環虚脱/死亡
```

#### EV-HZ002-001(HZ-002 過少投与)

```
[原因] ウォッチドッグタイマの誤発火 → 制御タスク停止
  ↓
[中間状態] 昇圧剤投与が無告知で停止
  ↓
[ハザード状況] 生命維持薬剤の途絶 (HZ-002)
  ↓
[危害] ショック状態の悪化 → 死亡
```

#### EV-HZ004-001(HZ-004 閉塞検知失敗 → アラーム失敗)

```
[原因] 閉塞検知アルゴリズムの閾値誤設定 (HZ-004)
  + アラームタスクのデッドロック (HZ-005)
  ↓
[中間状態] 閉塞発生も検知・通知なし
  ↓
[ハザード状況] 薬剤途絶の放置
  ↓
[危害] 致死的転帰(昇圧剤等の場合)
```

#### EV-HZ006-001(HZ-006 計算誤り)

```
[原因] 体重換算投与量の計算で整数オーバーフロー発生
  ↓
[中間状態] 内部値が負数・極小値にラップアラウンド
  ↓
[ハザード状況] 予期せぬ小流量注入 or ゼロ流量 (HZ-002 との組合せ)
  ↓
[危害] 投与不足 → 治療失敗
```

#### EV-HZ007-001(HZ-007 再起動時二重投与)

```
[原因] 停電→復電時、停止済み注入を「中断中」と誤認識し自動再開
  ↓
[中間状態] すでに完了した投与が再実行される
  ↓
[ハザード状況] 二重投与 (HZ-001 へ連鎖)
  ↓
[危害] 薬剤過量 → 致死的転帰
```

全事象シーケンスは SRS/SAD 完成に合わせて拡充する。

## 5. リスク推定(ISO 14971 箇条 5.5)

### 5.1 重大度と発生可能性の評価基準

#### 重大度(Severity)

| 段階 | 区分 | 例 |
|------|------|-----|
| S5 | 破局的 | 死亡、永続的重傷 |
| S4 | 重大 | 重傷、入院加療を要する有害事象 |
| S3 | 重度 | 医療介入を要する傷害 |
| S2 | 軽度 | 一時的で軽微な傷害 |
| S1 | 無視可能 | 一時的な不快感のみ |

#### 発生可能性(Probability)

| 段階 | 区分 | 頻度の目安 |
|------|------|----------|
| P5 | 頻繁 | > 10⁻³ / 使用 |
| P4 | 可能性大 | 10⁻³ 〜 10⁻⁴ |
| P3 | 時々 | 10⁻⁴ 〜 10⁻⁵ |
| P2 | 稀 | 10⁻⁵ 〜 10⁻⁶ |
| P1 | ほとんどない | < 10⁻⁶ |

> **本プロジェクトにおける発生可能性の推定根拠**:仮想機器のため実運用データがない。公開されている輸液ポンプのリコール情報(FDA MAUDE、PMDA 医療機器不具合情報)、AAMI 関連レポート、業界統計を参照して推定する。推定値は SRS/SAD/試験で具体の RCM が決まった後に再評価する。

### 5.2 リスクマトリクス

|        | S1 | S2 | S3 | S4 | S5 |
|--------|-----|-----|-----|-----|-----|
| **P5** | 中 | 高 | 高 | 許容外 | 許容外 |
| **P4** | 低 | 中 | 高 | 高 | 許容外 |
| **P3** | 低 | 中 | 中 | 高 | 高 |
| **P2** | 無視 | 低 | 中 | 中 | 高 |
| **P1** | 無視 | 低 | 低 | 中 | 中 |

**許容方針:**

- **許容外:** 必ずリスクコントロール実施。リリース不可。
- **高:** ALARP 原則適用。追加コントロールの検討必須、技術的・経済的に合理的な範囲で低減。
- **中:** 必要に応じて低減。ALARP 判断を記録。
- **低 / 無視:** 原則受容。ただし RCM 実装による悪化がないかを確認。

### 6.2 コントロール種別の優先順位(ISO 14971 箇条 7.1)

ISO 14971 の階層に厳密に従う:

1. **設計による本質的安全**(最優先)
   - ハザード原因の構造的排除(例: 型システムによる単位混在防止、Decimal による浮動小数誤差回避、状態機械による不能状態排除)
2. **製品自体又は製造工程における保護手段**
   - 冗長チェック、ウォッチドッグ、自己診断、フェイルセーフ停止、アラーム
3. **安全に関する情報**(最低優先)
   - 取扱説明書、画面上の警告、ラベル

本プロジェクトではソフトウェアのみで RCM を実装するため、ハードウェア外部手段は考慮しない(SSC-VIP-001 方針)。

### 8.2 ベネフィットとリスクの対比

**初期評価(Phase 3 時点):**

ベネフィット:
- 処方に従った薬剤を精密な流量で持続投与でき、手動投与に比べて患者安全・医療従事者の負担軽減に大きく寄与する。
- 持続注入が必要な昇圧剤・インスリン・鎮静剤等は輸液ポンプ無しでは実質的に実施困難な治療が多数存在する。

残留リスク(想定):
- 全 RCM が実装・検証された場合、§6.1 の目標どおりであれば残留リスクはすべて「中」以下に収まる見込み。
- 実機ではハードウェア側独立 RCM(機械的流量制限機構等)の追加により更に低減可能だが、本プロジェクトは仮想のためソフトウェア内 RCM のみで達成する。

**最終判定:**

初回リリース前に §8.1 のチェックリストを充足した段階で本節を更新し、判定記録を残す。

## 10. 製造後の情報収集及びリスク再評価(ISO 14971 箇条 10)

**本プロジェクトは仮想のため実製造後監視は運用しない** が、SMP-VIP-001 §4.1 に従い以下の枠組みを維持する。

### 10.1 情報源

- GitHub Issues(セルフフィードバック窓口)
- SOUP 脆弱性情報(pip-audit、GitHub Advisory Database、CVE/NVD)
- Python 公式 Security Announcements
- Dependabot アラート
- (実製品展開時追加)顧客フィードバック、FDA MAUDE、PMDA 医療機器不具合情報、同種機器のリコール情報

### 10.2 収集頻度

- 常時: GitHub Issues、CI 失敗、Dependabot
- コミットごと: pip-audit スキャン
- 月次: SOUP リリースノート確認
- 四半期: LTS 系更新判断(Python ランタイム等)

### 10.3 再評価のトリガ

以下のいずれかが発生した場合、本 RMF の該当箇所を更新し、バージョンを改訂する:

- 新規ハザードの識別
- 既存ハザードの発生頻度・重大度が推定を超えた
- RCM の無効化・劣化(試験 Fail、実運用での機能不全)
- SOUP の重大脆弱性・不具合の公表(CVSS ≥ 7.0 または利用箇所に影響)
- ソフトウェア変更(SRMP §7.4 の影響解析で「新規ハザード出現」判定)

### 10.4 処置

1. 本 RMF 更新(該当節に追記、改訂履歴に記録)
2. 関連ドキュメント(SRS / SAD / SDD / 試験計画・報告)を同一 PR 単位で更新
3. SRMP §7.4 に従いソフトウェア変更のリスクマネジメントを実施
4. 必要に応じて SPRP(PRB)→ SCMP(CR)→ 実装 → 検証 → リリース
