## 1. 目的

本書は、IEC 62304 箇条 4.3 に基づき、Virtual Infusion Pump Control Software(VIP-CTRL) のソフトウェア安全クラスを決定し、その根拠を記録する。本書は、ソフトウェア開発計画書(SDP)、ソフトウェア要求仕様書(SRS)、ソフトウェアリスクマネジメント計画書(SRMP) 等、後続のすべてのライフサイクル成果物の前提となる。

## 3. ソフトウェアシステムの概要

- **ソフトウェア名称:** Virtual Infusion Pump Control Software (VIP-CTRL)
- **意図する使用目的:** 処方に従って薬液を設定速度・設定量で患者に持続的に注入するための輸液ポンプ機構を制御する。設定・監視・アラーム機能を提供する。(本プロジェクトでは当該機能を PC 上のシミュレータで再現する)
- **動作環境:** 本来は医療機器組込み環境(マイコン + 液晶タッチパネル + ポンプ機構 + センサー)を想定するが、本プロジェクトでは PC 単体で動作する仮想環境。ハードウェア I/O(モータ駆動、圧力センサ、液滴センサ、UI 入力等)はソフトウェアモデルで置換する。
- **ユーザ・操作者:** 医療従事者(医師・看護師・薬剤師)。患者は被操作対象であり操作者ではない。
- **主な機能:**
  - 流量・投与量・投与時間の設定と保持
  - 定常注入・ボーラス投与・段階的投与の制御
  - 閉塞・気泡・薬液切れ・バッテリ低下等の異常検知とアラーム
  - 注入履歴・アラーム履歴のロギング
  - 操作者による設定変更・停止・再開

## 4. ソフトウェア安全クラスの定義(IEC 62304 箇条 4.3 より)

| クラス | 定義 |
|--------|------|
| A | 傷害又は健康被害はあり得ない |
| B | 重傷には至らない傷害があり得る |
| C | 死亡又は重傷があり得る |

クラス決定は、**ハードウェア外部リスクコントロール手段を考慮しない前提** で、ソフトウェアが寄与し得るハザード状況から生じ得る危害の重大度に基づく。

## 5. ハザード寄与解析の要約

輸液ポンプに関する公知のハザード(IEC 60601-2-24、FDA Infusion Pump Improvement Initiative、過去のリコール事例等)を参照し、本ソフトウェアが寄与し得るハザード状況を抽出した。

| ハザード ID | ハザード状況 | 想定される危害 | 重大度(システムレベル) | ソフトウェア寄与有無 |
|-----------|-----------|------------|---------------------|--------------------|
| HZ-001 | 過量投与(設定値を超える流量・積算量での注入) | 薬剤過量による循環虚脱・中毒・死亡(例: 麻薬、インスリン、高濃度カリウム、昇圧剤) | 致命的 | あり |
| HZ-002 | 過少投与・意図しない注入停止 | 生命維持に必須の薬剤が途絶えることによる急性増悪・死亡(例: 昇圧剤、抗不整脈薬) | 致命的 | あり |
| HZ-003 | 誤薬剤プロファイル適用(薬剤ライブラリ取り違え) | 薬剤に対して不適切な流量で投与されることによる重傷・死亡 | 致命的 | あり |
| HZ-004 | 閉塞・気泡・薬液切れの検知失敗 | 薬剤途絶に気付かず放置される/空気塞栓による脳梗塞・心停止 | 致命的 | あり |
| HZ-005 | アラーム失敗(アラームが鳴らない・無効化される・誤消音) | 異常状態が放置され、上記ハザードが顕在化する | 致命的 | あり |
| HZ-006 | 流量・ボーラス投与量の計算誤り(単位変換誤り、丸め誤差、オーバーフロー) | 想定の 10 倍・100 倍投与等による重篤な中毒・死亡 | 致命的 | あり |
| HZ-007 | 電源喪失・再起動時の不定動作(中断した注入の誤った再開、設定喪失) | 意図しない二重投与・投与中断 | 致命的 | あり |
| HZ-008 | 操作者による意図しない操作の確定(デバウンス不良、誤タッチの確定) | 意図しない用量で投与される | 重大〜致命的 | あり |

## 6. 最悪シナリオにおけるソフトウェア起因の危害

ハードウェア外部リスクコントロール手段を仮に無しとした場合、ソフトウェア故障により生じ得る最悪シナリオを評価する。

| シナリオ | 想定される危害 | 重大度 |
|---------|------------|-------|
| 用量計算ロジックの不具合により設定の 10 倍量を注入 | 死亡(例: インスリン 10 倍投与による低血糖昏睡) | 死亡又は重傷 |
| 閉塞検知アルゴリズムの不具合により昇圧剤投与停止を検知できず放置 | ショック状態の悪化・死亡 | 死亡又は重傷 |
| アラームタスクのデッドロックにより異常時にアラームが発報しない | 異常放置による致死的転帰 | 死亡又は重傷 |
| 気泡検知の失敗により大量の空気が血管内に注入 | 空気塞栓による脳梗塞・心停止 | 死亡又は重傷 |
| 再起動時に停止済みの注入を誤って再開 | 重複投与による薬剤過量・死亡 | 死亡又は重傷 |

## 7. 安全クラスの決定

### 7.1 決定結果

**本ソフトウェアの安全クラス: C**

### 7.2 根拠

以下の観点から、本ソフトウェアはクラス C に該当する。

- [x] 故障時に死亡又は重傷が生じ得るハザード状況に寄与する(HZ-001〜HZ-008 のすべて)
- [x] ハードウェア外部リスクコントロール手段(機械的流量制限、二次独立ハードウェア等)のみではソフトウェア故障に起因する過量/過少投与を完全に緩和できない。流量・積算量の制御ロジック自体がソフトウェア内にあり、一次的なリスクコントロール手段もソフトウェア(冗長チェック、ウォッチドッグ、自己診断)で実装する必要がある
- [x] 寄与原因がソフトウェア内部(制御アルゴリズム、アラーム管理、UI 入力確定、状態遷移)にあり、リスクコントロール手段(RCM)もソフトウェアで実装する必要がある
- [x] 公知事例(過去の輸液ポンプリコール、FDA Infusion Pump Improvement Initiative)において、ソフトウェア起因の死亡・重傷事例が多数報告されている

### 7.3 下位クラスへの分割検討(箇条 5.3.5 関連)

高クラスの RCM を実装する項目と、他の項目とを分離することで、非 RCM 項目の安全クラスを下げることを検討する。本時点ではアーキテクチャ未確定のため、想定される分割候補とその方針を記録する。詳細な決定はアーキテクチャ設計書(箇条 5.3)にて行い、本書を改訂する。

| 項目(仮称) | 分離の実現性 | 分離後のクラス(想定) | 決定 |
|------|-----------|------------|------|
| 流量制御コア(モータ指令、積算量管理) | 分離不可(安全コア) | C | クラス C のまま |
| アラーム管理(検知→通知→確認) | 分離不可(安全機能) | C | クラス C のまま |
| 用量計算・単位換算 | 分離不可(計算結果が安全に直結) | C | クラス C のまま |
| 注入ログ表示(読み取り専用 UI) | 可(プロセス/モジュール分離により制御系と独立化可能) | B もしくは A | アーキテクチャ設計時に再評価 |
| 設定画面 UI(入力) | 入力確定後の検証を安全コア側で行えば表示層のみ分離可能 | B | アーキテクチャ設計時に再評価 |
| 使用統計の外部送信機能 | 可(制御系と完全分離) | A | アーキテクチャ設計時に再評価 |

> 分離を採用する場合は、IEC 62304 箇条 5.3.5 に従い、分離の根拠(データフロー・制御フロー・失敗伝播経路が遮断されていること)をアーキテクチャ設計書に文書化すること。

## 8. クラス変更の要件

以下の事由が発生した場合、クラスを再評価する。

- システムリスク解析の結果変更
- 新たなハザード識別
- 意図する使用目的の変更(例: 対象薬剤カテゴリの追加、対象患者群の拡大)
- ハードウェア外部リスクコントロール手段の追加・変更(例: 独立した機械式流量制限機構の追加)
- 分離設計の変更(アーキテクチャ変更に伴う項目別クラスの再決定)
- 本プロジェクトの「仮想」前提の変更(仮に実機展開を検討する場合)
